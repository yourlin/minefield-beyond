# Product Requirements Document - 高挑战性扫雷游戏

**Author:** Linyesh
**Date:** 2026-04-27

## Executive Summary

本产品是一款面向益智游戏爱好者的 Web 端高挑战性扫雷变体游戏，使用 TypeScript + 原生 Canvas API 构建，Vite 作为开发服务器和生产构建工具。产品通过引入非传统棋盘拓扑和创新信息机制，彻底重构经典扫雷的推理体验。

核心设计理念：在不断变化的规则和棋盘形态下，始终保证每一关可解。地雷在玩家首次点击时运行时随机放置，首次点击位置及其邻居保证安全（无雷区），关卡文件仅存储棋盘形状和地雷数量，不预设地雷位置。

游戏采用 30 关线性递进制，分为 3 个拓扑类别（Hex 六边形、Torus 环面、Triangle 三角形），每类 10 关。六边形关卡通过 cube coordinates 生成多种棋盘形状（矩形、菱形、六角形、三角形、倒三角、十字等），每一关引入新的视觉或规则元素，从第一关起就制造新奇感，并在整个游戏过程中持续释放。关卡数据以自定义二进制格式（MSWP）存储，配套关卡阅读器工具和离线可解性验证器。玩家进度通过 localStorage 持久化保存。

### What Makes This Special

市面上的扫雷变体通常走两条路：换皮肤不换玩法，或引入随机性将推理变成赌博。本产品的独特定位在于——**规则在变，逻辑永远成立**。

具体体现在两个核心维度：
- **棋盘拓扑创新**：六边形（cube coordinates 生成多种形状）、环面（边缘环绕）、三角形等非矩形网格，每种拓扑改变邻居关系和数字语义，迫使玩家重建空间推理模型
- **信息机制变革**：模糊提示（范围值）、延迟揭示等机制改变信息获取方式，经过精心设计确保组合推理后关卡可解

这种"逻辑推理 + 不断变化的规则空间 + 运行时随机地雷放置"的组合，在现有扫雷类产品中尚无先例。

## Success Criteria

### User Success

- 玩家从第一关起就能感受到"这不是普通扫雷"的新奇感
- 每一关引入的新元素（拓扑或信息机制）能在 30 秒内被玩家理解规则
- 分层通关时间：教学关（1-10）≤ 2 分钟，中段关卡（11-20）≤ 5 分钟，硬核关（21-30）≤ 8 分钟
- 支持无限重试，重试时保留玩家已有的推理标记，降低重试摩擦
- 首次通关率目标：教学关（1-10）≥ 90%，中段关卡（11-20）40%-60%，硬核关（21-30）≥ 20%
- 每关弃关率（玩家放弃不再尝试）≤ 40%
- 30 关难度曲线在每个拓扑类别内平滑递进，跨类别切换时提供过渡引导
- 每一关至少包含一个设计意图上的"啊哈时刻"——玩家通过推理顿悟新规则的关键瞬间
- 首次遇到新机制时提供交互式引导，之后提供可随时查看但不强制弹出的规则卡片
- 踩雷失败时提供清晰的视觉反馈和死因回顾，玩家能理解自己的错误在哪里

### Business Success

- 完成全部 30 关的可玩版本，可部署到 Web 并分享链接
- 在独立游戏社区（如 itch.io）发布并获得至少 50 次游玩
- 收到至少 5 条来自非作者本人的有效反馈
- 项目代码结构清晰，可作为个人作品集展示

### Technical Success

- 验证器输入包括棋盘拓扑定义、形状参数和地雷数量
- 每种拓扑类型有正式的邻接规范文档，配套单元测试覆盖边界情况
- 二进制关卡文件格式包含 magic number + 版本字节，关卡阅读器对异常文件给出明确错误
- 关卡阅读器能正确加载、可视化和模拟所有 10 个关卡
- 游戏在 Chrome 桌面端和移动端流畅运行，交互响应延迟 < 100ms，帧率 ≥ 30fps
- 每关内存使用不超过 50MB，避免浏览器 OOM

## User Journeys

### 旅程一：小明 — 益智游戏爱好者的首次体验

小明是一个 28 岁的程序员，业余时间喜欢玩逻辑解谜游戏。他在 itch.io 上看到了这款游戏的截图——一个六边形的扫雷棋盘，立刻被吸引点了进去。

打开游戏，关卡选择界面显示 30 个关卡，分为 Hex、Torus、Triangle 三个拓扑类别，只有第 1 关解锁。他点进去，看到一个小型六边形矩形网格。一个简短的交互式引导高亮了第一步，他立刻意识到——每个格子有 6 个邻居而不是 8 个。"有意思。"他花了 1 分钟通关，解锁了第 2 关。

到了第 3 关，棋盘形状变成了菱形——由 cube coordinates 生成的对称图案。他注意到边缘格子的邻居数量不同，需要重新思考推理策略。第一次踩雷了——死因回顾显示了他忽略的一个关键约束。"啊，原来如此。"重试时他保留了之前的标记，很快通关。

到了第 11 关，进入 Torus 拓扑类别——棋盘边缘相连，没有安全角落。他花了 3 次尝试才适应这个概念，但每次失败后的回顾都让他更接近答案。通关那一刻他忍不住说了声"漂亮"。

### 旅程二：Linyesh — 关卡设计者的工作流

Linyesh 正在设计第 6 关——一个中等大小的六边形菱形棋盘。他在关卡生成脚本中定义了棋盘拓扑类型、cube coordinates 形状参数和地雷数量，然后运行可解性验证器。验证器模拟首次点击后的运行时地雷放置，验证在该棋盘形状和地雷密度下关卡可解。

验证通过后，生成脚本将关卡数据编码为 MSWP 二进制文件。他用关卡阅读器加载生成的文件，可视化检查棋盘布局、邻接关系和形状对称性。一切正确，他将关卡文件保存到项目中。

### 旅程三：小明 — 卡关与放弃边缘

小明打到第 25 关——三角形拓扑加模糊提示。他连续失败了 5 次，每次都在同一个区域犯错。他开始怀疑这关是不是有 bug。但死因回顾清楚地显示了他每次的推理错误。第 6 次尝试，他换了一个起点，终于找到了突破口。"这关设计得真狠，但确实公平。"

## Innovation & Novel Patterns

### Detected Innovation Areas

**1. 拓扑驱动的推理重构（核心创新）**

传统扫雷建立在矩形网格的固定邻接关系上，所有推理策略都围绕"8邻居"展开。本产品将棋盘拓扑本身变成了变量——六边形（6邻居）、三角形（3/12邻居）、环面（无边界）。这从根本上改变了推理的数学基础。

**2. Cube Coordinates 棋盘形状生成系统**

六边形关卡通过 cube coordinates（q, r, s 其中 q+r+s=0）在数学空间中定义棋盘形状，支持矩形、菱形、六角形、三角形、倒三角、十字等多种对称形状，然后转换为 odd-r offset 坐标存储。这保证了数学上的精确对称性。

**3. 运行时地雷放置与首次点击安全区**

地雷不预存在关卡文件中，而是在玩家首次点击时运行时随机放置。首次点击位置及其所有邻居保证无雷，为玩家提供安全的起始推理区域。关卡文件仅存储棋盘形状和地雷数量。

**4. 信息机制与可解性的矛盾统一**

模糊提示和延迟揭示在直觉上会增加推理难度。本产品的创新在于：即使在信息不完全的条件下，通过精心设计的约束组合，仍然可以保证关卡可解。

**5. 二进制关卡格式（MSWP）作为可扩展平台**

将关卡数据从代码中解耦为独立的二进制格式（MSWP header + 版本字节），为未来的社区关卡编辑器和分享平台奠定基础。

### Market Context & Competitive Landscape

- **换皮类**（如 Minesweeper Genius）：改变视觉风格但保留矩形网格
- **随机化类**（如 Demoncrawl）：引入 roguelike 元素，牺牲确定性
- **六边形类**（如 Hexcells）：最接近的竞品，但仅限六边形拓扑，无信息机制变革

本产品的差异化：多拓扑 × 多信息机制 × 运行时地雷放置 × cube coordinates 形状系统的组合，无直接竞品。

### Validation Approach

- 可解性验证器在所有拓扑和信息机制组合下验证关卡可解性 → 创新成立
- 教学关达到 90% 首次通关率 → 新拓扑的可学习性成立
- 硬核关弃关率 ≤ 40% → 高难度下的公平感知成立

## Functional Requirements

### 棋盘拓扑系统

- FR1: 游戏引擎可以通过 Canvas API 渲染六边形网格棋盘（pointy-top odd-r offset）并正确计算 6 邻居邻接关系
- FR2: 六边形关卡通过 cube coordinates（q, r, s 其中 q+r+s=0）生成多种棋盘形状：矩形、菱形、六角形、三角形、倒三角、十字
- FR3: 游戏引擎可以渲染环面拓扑棋盘（矩形），边缘格子与对侧边缘格子正确相连
- FR4: 游戏引擎可以渲染三角形网格棋盘（矩形）并正确计算 3/12 邻居邻接关系
- FR5: 所有拓扑类型通过统一的抽象接口（ITopologyGraph）提供邻接查询能力（给定 cell，返回其所有邻居）
- FR6: 新增拓扑类型只需实现统一接口并注册到 TopologyRegistry，不需要修改现有拓扑代码
- FR7: 每种拓扑类型定义从逻辑坐标到屏幕坐标的映射规则（含环面的 wrap-around 视觉提示）
- FR8: 不同拓扑下的点击/触摸坐标可以正确映射到对应的逻辑 cell（拓扑适配的 hit-test）
- FR9: Canvas 渲染器根据棋盘大小自动计算缩放因子，所有绘制尺寸（格子、字体、线宽）乘以缩放因子

### 信息机制系统

- FR10: 玩家翻开格子后可以看到模糊提示（数字范围值，如"1-3"），表示周围地雷数在该范围内
- FR11: 玩家翻开格子后，数字信息可以延迟指定回合数后才显示
- FR12: 每种信息机制有形式化的语义定义：模糊提示为"真实值 ± 偏移量"的范围；延迟揭示为"翻开后延迟 N 步才显示数字"
- FR13: 多种信息机制同时作用于同一关卡时，其组合行为有明确定义（独立作用，不叠加）
- FR14: 延迟揭示期间，格子显示明确的"等待中"视觉状态

### 游戏核心交互

- FR15: 玩家首次点击格子时，系统运行时随机放置地雷，首次点击位置及其所有邻居保证无雷
- FR16: 玩家可以通过鼠标左键（桌面端）或点击（移动端）翻开未标记的格子
- FR17: 玩家可以通过鼠标右键（桌面端）或长按（移动端）标记/取消标记疑似地雷的格子
- FR18: 玩家翻开安全格子时，系统自动展开相邻的无雷区域
- FR19: 玩家踩雷时，游戏立即结束并显示失败反馈
- FR20: 玩家在移动端可以通过缩放和平移手势操作棋盘
- FR21: 玩家可以撤销最近一次标记操作（防误操作，不可撤销翻开操作）
- FR22: 游戏提供暂停功能，暂停时棋盘被遮蔽，计时器停止
- FR23: 玩家切换浏览器标签页或手机来电时，游戏自动暂停

### 视觉与音效反馈

- FR24: 每种格子状态（未翻开、已翻开、已标记、延迟等待中、模糊提示）有明确区分的视觉表现
- FR25: 格子状态转换时有过渡动画（翻开、标记、踩雷）
- FR26: 游戏提供基础音效反馈（Web Audio API）：翻开格子、标记地雷、踩雷失败、通关成功
- FR27: 玩家可以在设置中开关音效

### 关卡流程管理

- FR28: 玩家可以在关卡选择界面查看所有 30 个关卡（分 3 个拓扑类别展示）及其解锁状态
- FR29: 玩家通关当前关卡后，下一关自动解锁
- FR30: 玩家可以重玩任何已解锁的关卡
- FR31: 每关显示分层计时器（教学关 2 分钟 / 中段 5 分钟 / 硬核 8 分钟上限）
- FR32: 玩家可以无限次重试当前关卡，重试时保留已有的推理标记
- FR33: 玩家通关后可以看到通关动画和完成反馈
- FR34: 玩家的关卡解锁进度、最佳通关时间和尝试次数通过 localStorage 自动持久化保存（`mswp_` 前缀），关闭浏览器后不丢失
- FR35: 关卡选择界面显示每关的最佳通关时间和尝试次数
- FR36: 每个关卡有明确的状态生命周期：未开始 → 进行中 → 成功/失败，状态转换触发对应的 UI 和逻辑响应
- FR37: 开发模式（`?dev` URL 参数）解锁所有关卡

### 教学与引导系统

- FR38: 玩家首次遇到新拓扑类型时，系统提供交互式引导（如高亮第一步操作）
- FR39: 玩家首次遇到新信息机制时，系统提供交互式引导说明新规则
- FR40: 玩家可以在游戏中随时查看当前关卡的规则卡片（不强制弹出）
- FR41: 规则卡片展示当前关卡的拓扑类型和信息机制规则

### 失败反馈与死因回顾

- FR42: 玩家踩雷后可以看到包含爆炸动画和地雷位置高亮的失败反馈
- FR43: 玩家踩雷后可以进入死因回顾界面，查看完整的地雷分布
- FR44: 死因回顾界面标注玩家的推理错误位置
- FR45: 死因回顾界面可以展示正确的解题路径（推理链）
- FR46: 玩家可以从死因回顾界面直接重试当前关卡

### 关卡数据与工具链

- FR47: 关卡数据以自定义二进制格式（MSWP）存储，包含 magic number（0x4D 0x53 0x57 0x50）和版本字节（0x01）
- FR48: 二进制格式存储棋盘拓扑类型、形状参数、格子定义和地雷数量；地雷位置不预存，运行时放置
- FR49: 关卡阅读器可以加载 MSWP 二进制关卡文件并可视化展示棋盘布局
- FR50: 关卡阅读器可以展示邻接关系和信息机制参数
- FR51: 关卡阅读器可以模拟游戏过程（逐步翻开格子查看结果）
- FR52: 关卡阅读器对损坏、截断或版本不匹配的文件给出明确错误信息
- FR53: 离线可解性验证器可以验证给定关卡在其特定棋盘形状和地雷数量配置下可解
- FR54: 可解性验证器使用约束传播 + 回溯算法
- FR55: 可解性验证器在验证失败时输出具体原因
- FR56: 游戏运行时、关卡阅读器和可解性验证器共享同一套核心代码库（core/），包含拓扑邻接计算和二进制格式解析

### 部署与分发

- FR57: 游戏可以通过 Vite 构建为静态 Web 资源并部署到任意静态服务器
- FR58: 游戏在 Chrome 桌面端和移动端浏览器上可正常运行
- FR59: 游戏提供一个简单的 HTML 落地页（含游戏描述和启动按钮）
- FR60: 关卡文件（.mswp）作为静态资源打包在应用内，通过 fetch 加载

## Non-Functional Requirements

### Performance

- NFR1: 玩家点击格子到视觉反馈的响应延迟 < 100ms（所有拓扑类型）
- NFR2: 游戏帧率在所有关卡中保持 ≥ 30fps
- NFR3: 游戏首次加载时间 < 3s（Chrome 桌面端，含 Canvas 渲染代码和关卡资源）
- NFR4: 关卡切换时间 < 1s（包括二进制文件解析和棋盘渲染）
- NFR5: 自动展开无雷区域的连锁反应在 200ms 内完成视觉呈现（通过动画帧计数验证）
- NFR6: 每关内存使用不超过 50MB，避免浏览器 OOM
- NFR7: 离线可解性验证器对单个关卡的验证时间 < 30s

### Accessibility

- NFR8: 格子状态使用高对比度配色，满足 WCAG 2.1 AA 级对比度标准（4.5:1）
- NFR9: 数字和范围提示字体大小在移动端不小于 14px 等效
- NFR10: 不依赖纯色彩区分关键信息（格子状态同时使用形状或图标辅助区分）

### Maintainability

- NFR11: 拓扑邻接计算、二进制格式解析作为独立共享模块，运行时/阅读器/验证器共用
- NFR12: 新增一种拓扑类型只需实现统一接口，不需要修改现有拓扑代码
- NFR13: 新增一种信息机制只需实现规则定义接口，不需要修改验证器核心逻辑
- NFR14: 代码遵循 TypeScript 严格模式，启用 strict 编译选项

### Reliability

- NFR15: localStorage 进度数据损坏时，游戏降级为全部关卡锁定状态（第 1 关除外），不崩溃
- NFR16: 二进制关卡文件（MSWP）加载失败时，显示明确错误信息并返回关卡选择界面，不崩溃
- NFR17: 浏览器标签页切换或手机来电导致的中断不丢失当前关卡状态
